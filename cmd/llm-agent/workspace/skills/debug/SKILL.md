# 问题诊断

## 适用场景

- 用户要求排查服务异常、接口报错、进程退出、端口冲突
- 需要查看本地或远程日志、进程状态、监听端口
- 需要先定位根因，再决定是否修复

## 必须遵守

- 先确认目标服务、日志路径、时间范围，再执行命令
- 先看最近日志，再按关键词或时间过滤，不要一开始就抓整份大日志
- 远程主机必须使用当前任务上下文里的真实目标地址，不能硬编码固定服务器
- 能通过日志或状态定位问题时，先定位再给修复建议

## 推荐流程

1. 确认要排查的是本地还是远程环境，以及服务名、日志路径和异常时间段。
2. 用 `tail` 看最近日志，例如：
   - 本地：`tail -200 /data/logs/app.log`
   - 远程：`ssh <user>@<host> 'tail -200 /data/logs/app.log'`
3. 再用 `grep`、时间串、错误关键字缩小范围，例如：
   - `ssh <user>@<host> 'grep -i error /data/logs/app.log | tail -50'`
   - `ssh <user>@<host> 'grep 2026-03-17 /data/logs/app.log | grep -i error'`
4. 如有必要，再补查进程、端口、磁盘、权限等状态。
5. 根据证据给出根因判断和修复建议。

## 工具选择规则

- 日志、进程、端口、文件权限检查统一用 `Bash`
- 本地问题直接执行本地命令；远程问题通过 `ssh <user>@<host>` 包装
- 只有已经拿到足够日志证据后，才进入代码或配置修复阶段

## 禁止行为

- 把固定 IP 或固定账号写进通用排障流程
- 一次输出超大日志，导致上下文被无效文本占满
- 没看日志就直接给“可能原因”列表
- 在未知日志路径时凭空猜路径并据此下结论

## 示例

- “线上接口 500 了，帮我看最近 10 分钟日志”
  先确认服务和日志路径，再用 `tail` 和 `grep` 按时间窗口过滤
- “服务起不来，怀疑端口被占”
  先查日志，再查进程和监听端口，最后给出冲突进程或启动失败原因
